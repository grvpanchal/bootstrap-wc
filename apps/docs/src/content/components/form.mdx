`<bs-form>` is a thin wrapper around a native `<form>`. It exists for
one purpose: give the form-control components an ancestor `<form>` and
hoist submit into a single cancellable `bs-submit` event whose
`detail.formData` is a fresh snapshot.

## How autofill works here

Browser autofill heuristics and password managers (Chrome, Safari,
Firefox, 1Password, Bitwarden, …) walk the **light-DOM** tree looking
for `<input name="…" autocomplete="…">` inside a `<form>`, and anchor
their autofill UI to the focused control. Anything inside a shadow root
is invisible to them.

So our form controls — `bs-input`, `bs-textarea`, `bs-select`,
`bs-range`, `bs-form-check` — render their native control directly
into **light DOM**, not into a shadow root. This is the same approach
Ionic, Adobe Spectrum, Lion, and Carbon's web-component libraries take
for form controls. `bs-form` simply provides the `<form>` ancestor
those native controls live inside.

End result: open any of these forms on a device with saved
credentials / address / contact info and the browser's autofill chip
appears on the right inputs — exactly as it would for a hand-written
HTML form. Each form below intercepts submit and logs the `FormData`
to the console instead of navigating.

## Sign in

Classic two-field form. `autocomplete="username"` on the identifier and
`autocomplete="current-password"` on the password let every password
manager (browser-native, 1Password, Bitwarden, …) volunteer a saved
login.

<Example code={`<bs-form id="ex-login" novalidate>
  <div class="mb-3">
    <bs-form-label for="login-user">Email or username</bs-form-label>
    <bs-input id="login-user" name="username" type="text" autocomplete="username" required></bs-input>
  </div>
  <div class="mb-3">
    <bs-form-label for="login-pw">Password</bs-form-label>
    <bs-input id="login-pw" name="password" type="password" autocomplete="current-password" required></bs-input>
  </div>
  <div class="mb-3 d-flex justify-content-between align-items-center">
    <bs-form-check name="remember" value="1" label="Remember me" autocomplete="off"></bs-form-check>
    <a href="#">Forgot password?</a>
  </div>
  <bs-button type="submit" variant="primary">Sign in</bs-button>
</bs-form>
<script>
  document.getElementById('ex-login')?.addEventListener('bs-submit', (ev) => {
    ev.preventDefault();
    console.log('sign-in:', Object.fromEntries(ev.detail.formData.entries()));
  });
</script>`} />

## Shipping address

and most password managers all maintain saved addresses. Uses the
[WHATWG address autocomplete tokens](https://html.spec.whatwg.org/multipage/form-control-infrastructure.html#autofill):
`name`, `street-address`, `address-line1` / `-2`, `address-level2`
(city), `address-level1` (state / province), `postal-code`, and
`country` / `country-name`.

## Validation

`<bs-form>` applies Bootstrap's `.needs-validation` class to the inner
form and toggles `.was-validated` after the first submit attempt
(unless `novalidate` is set). Combined with `bs-input[required]` /
`[pattern]` / `[minlength]` etc. you get the full `:valid` / `:invalid`
styling with no extra JS.

<Example code={`<bs-form id="ex-validate">
  <bs-form-label for="val-user">Username</bs-form-label>
  <bs-input id="val-user" name="username" required minlength="3" autocomplete="username"></bs-input>

  <bs-form-label for="val-pw" class="mt-3">Password</bs-form-label>
  <bs-input id="val-pw" name="password" type="password" required minlength="8" autocomplete="current-password"></bs-input>

  <div class="mt-3">
    <bs-button type="submit" variant="primary">Submit</bs-button>
  </div>
</bs-form>`} />

## API

### Attributes

| Name | Type | Default | Description |
|---|---|---|---|
| `action` | `string` | — | Forwarded to the inner `<form>`. |
| `method` | `string` | — | Forwarded to the inner `<form>`. |
| `target` | `string` | — | Forwarded to the inner `<form>`. |
| `enctype` | `string` | — | Forwarded to the inner `<form>`. |
| `novalidate` | `boolean` | `false` | Disable native constraint validation. |
| `validated` | `boolean` | `false` | Applies `.was-validated`. Auto-set after first submit unless `novalidate`. |

### Events

| Event | Detail | Description |
|---|---|---|
| `bs-submit` | `{ formData, form, originalEvent }` | Fires on submit. Cancellable — `preventDefault()` stops the underlying form submission. |

### Methods & properties

- `reset()` — resets the inner form and clears `validated`.
- `checkValidity()` / `reportValidity()` — forwarded to the inner form.
- `formData` — fresh `FormData` snapshot.
- `nativeForm` — the underlying `<form>` element.

### How the form-control components are wired

Every form control in the library — `bs-input`, `bs-textarea`,
`bs-select`, `bs-range`, `bs-form-check` — renders its native control
(`<input>`, `<textarea>`, `<select>`) directly into **light DOM** rather
than into a shadow root. That choice is deliberate and matches what
Ionic, Adobe Spectrum, Lion, and Carbon's web-component libraries do
for form controls. The reason is twofold:

- Browser autofill predictors (Chrome, Safari, Firefox) walk the
  light-DOM tree to find inputs and decide what type they are. Inputs
  inside a shadow root are invisible to them.
- Autofill / password-manager UI is *anchored* to the focused control.
  A hidden mirror in a sibling never receives focus, so the autofill
  chip never appears even if the predictor saw it.

Because the native control is in light DOM, it's automatically inside
the nearest `<form>` ancestor (typically a `<bs-form>`) and participates
in `FormData` natively — no `ElementInternals` plumbing required. Each
component still exposes its `value` / `checked` / events on the host
for ergonomic property binding.

## Install

```sh
# as an npm package
npm install @bootstrap-wc/components
# or copy the source
npx bwc add form
```

## Usage

```ts
import '@bootstrap-wc/components/form';
```

```html
<bs-form action="/signup" method="post">
  <!-- form-associated <bs-*> controls go here -->
</bs-form>
```
