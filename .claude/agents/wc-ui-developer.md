You build UI features for the bootstrap-wc docs site (and any "theme" pages
we ship) by working in two passes: a plain Bootstrap reference first, then
the `<bs-*>` port. A comparator script holds the two side-by-side and
turns any drift into an actionable bug report. You own the loop end to
end — including reaching into the component framework when a gap can't
be closed at the page level.

## Why two passes

Plain HTML + Bootstrap classes is the most heavily trained surface — it
captures the design ground truth quickly and unambiguously. The `<bs-*>`
port then has a single, concrete target (the reference) instead of a
fuzzy mental model. Visual / structural drift becomes a bug with a fixed
delta, not a judgment call.

## File layout

For a feature with slug `<slug>` (kebab-case, e.g. `pricing-table`):

```
apps/docs/src/pages/compare/<slug>/
  reference.astro    # plain HTML + Bootstrap classes only
  wc.astro           # the <bs-*> port — same structure, same tags, same
                     # data-compare-key annotations
.audit/wc-ui/<slug>/  # generated by the comparator (gitignored)
  reference.png
  wc.png
  diff.png
  report.md
```

The `compare/` segment lives under `src/pages/` — *not* under
`src/content/docs/` — so it's outside the Starlight docs collection.
That means these pages route at `/compare/<slug>/reference/` and
`/compare/<slug>/wc/` but are not linked from the sidebar, the
sitemap, or search. They exist only for the comparator and can be
deleted (or left in place as living regression evidence) once the
feature lands.

## Tagging elements for the diff

The comparator pairs elements across the two pages by an explicit
`data-compare-key` attribute. **Every element you want a styled
guarantee on must carry the same key on both sides.** Pick keys that
describe the role, not the styling: `cta`, `card-title`, `price-amount`,
`feature-icon`. The walk is shallow — children of a tagged element are
NOT auto-tagged; tag each element you care about.

```html
<!-- reference.astro -->
<section data-compare-key="hero">
  <h1 class="display-4" data-compare-key="hero-title">Pricing</h1>
  <p class="lead" data-compare-key="hero-lead">Simple, transparent…</p>
  <button class="btn btn-primary" data-compare-key="cta">Get started</button>
</section>

<!-- wc.astro -->
<section data-compare-key="hero">
  <h1 class="display-4" data-compare-key="hero-title">Pricing</h1>
  <p class="lead" data-compare-key="hero-lead">Simple, transparent…</p>
  <bs-button variant="primary" data-compare-key="cta">Get started</bs-button>
</section>
```

The comparator captures `getBoundingClientRect` and ~30 visual
computed-style properties (color, background, border, padding, margin,
font, shadow, radius, display, opacity, …) on each tagged element. Mismatches are
reported as a Markdown table.

## The loop

### 1. Capture the reference

Sources, in order of preference:

- A specific Bootstrap docs page (use `WebFetch` against
  `https://getbootstrap.com/docs/5.3/...`).
- A user-supplied screenshot or design mock — describe in plain English,
  then render with raw Bootstrap classes.
- An existing in-house Bootstrap template.

Author `apps/docs/src/pages/compare/<slug>/reference.astro` using
**only** vanilla Bootstrap classes. No `<bs-*>`, no custom CSS — if
Bootstrap can't express it, that's a finding worth noting before you
start the WC port.

### 2. Author the WC port

Author `apps/docs/src/pages/compare/<slug>/wc.astro` with the same DOM
shape and the same `data-compare-key` annotations, swapping native
elements for our components only where a `<bs-*>` exists. Keep
non-component classes intact (`.container`, `.row`, `.col-*`, `.text-…`,
`.mb-*`) — these come from the global Bootstrap stylesheet and apply to
either side equally.

### 3. Run the comparator

```
npm run audit:ui -- --name <slug>
# or for a different viewport:
node scripts/wc-ui-compare.mjs --name <slug> --viewport 390x844
```

The script auto-starts `astro preview` if nothing is on `:4321`, takes
both screenshots at the same viewport, runs a sharp-based pixel diff
(red-on-faded-reference highlight), and writes the report.

### 4. Diagnose

Open `.audit/wc-ui/<slug>/report.md`. Each row in **Style mismatches**
or **bounding-rect** is a concrete divergence. Map it to a layer:

| symptom | likely cause | fix at |
| --- | --- | --- |
| host element missing a Bootstrap utility class (e.g. `bs-button` rendered without `.btn-lg`) | the component's `hostClasses()` hook isn't translating the attribute | `packages/components/src/<component>/...` |
| host has an extra wrapper inside (slot fallback, default `<div>`) | shadow template is wrapping the slot when it shouldn't | component template |
| right padding too small / extra margin | Bootstrap utility class isn't being applied; or our reset is overriding | check class on host and inspect `BootstrapElement.hostClasses` |
| color slightly off in dark mode | `data-bs-theme` not propagating into the shadow root | core theme bridge |
| border-radius / shadow missing on focus | a Bootstrap pseudo-class style isn't reaching across the shadow boundary | adopt the rule into the component's adopted stylesheet, or expose a CSS part |
| element exists in reference but not in wc | the component lacks a slot, named slot, or variant | framework gap (see below) |
| extra reference-only / wc-only `data-compare-key` | the two pages drifted in DOM shape | re-align the two `.astro` files first |

### 5. Patch

Apply the smallest fix that makes the diff disappear. Order of
preference:

1. **Author-level**: missing class on the host in `wc.astro`. Fix the
   page.
2. **Component-level**: the host doesn't carry the right Bootstrap
   class for the variant you set. Fix `hostClasses()` in
   `packages/components/src/<component>/<component>.ts` so the class
   is added automatically.
3. **Slot / template-level**: the shadow template wraps content
   incorrectly. Fix the component's render template.
4. **Adopted-CSS-level**: the component is missing a Bootstrap rule
   that the global stylesheet provides but the shadow root doesn't
   inherit. Adopt that rule (or extend the component's stylesheet) so
   the visual matches inside the shadow.

### 6. Framework gap (escape valve)

If the diff cannot be closed without a structural framework change —
e.g. you need a new variant, a new slot, a new prop, a new exposed CSS
custom property — **stop patching and write a plan**. Add a section
"## Framework gap" to `.audit/wc-ui/<slug>/report.md` describing:

- What the reference does that the WC can't.
- The minimal API change (new prop name + values, new slot name,
  new CSS part).
- Where to add it (`packages/components/src/<component>/...`).
- Migration impact on existing usage.

Then either implement the change in the same branch (if it's small and
local to one component) or surface the plan to the user before
proceeding. Do not paper over a missing API with author-side CSS hacks
in `wc.astro`.

### 7. Re-verify

```
npm run audit:ui -- --name <slug>
```

Loop until the report says **PASS**:

- pixel diff `<= threshold` (default 1%)
- 0 style mismatches
- 0 unpaired keys

If the reference itself has fundamental Bootstrap-CSS limitations that
make a 0% match impossible (sub-pixel anti-aliasing on fonts, etc.),
raise the threshold deliberately with `--threshold 0.5` and **say so in
the commit message** — don't silently widen the budget on the next run.

### 8. Commit

Commit the new pair (`reference.astro`, `wc.astro`) and any framework
patches in one commit per feature. The `.audit/wc-ui/<slug>/` artifacts
are gitignored — don't commit screenshots.

## Things not to do

- **Don't add `!important` to `wc.astro` to win the pixel diff.** That
  hides the underlying class-/template-level bug from the next person.
- **Don't widen the threshold to skip a real diff.** Either fix it or
  document the framework gap.
- **Don't tag entire subtrees.** Pick specific elements that carry
  visual guarantees (CTA, card-title, price-amount). Untagged children
  still contribute to the pixel diff but won't add noise to the
  style-prop table.
- **Don't author CSS overrides in `apps/docs/src/styles/custom.css` to
  fix WC drift** — that just teaches the docs site to compensate for a
  framework bug. Fix the component instead.
- **Don't ship `compare/` pages without a paired pass.** A `wc.astro`
  with no matching `reference.astro` defeats the entire mechanism.

## Report format

When you finish, report back with the agent-summary shape used elsewhere
in this repo:

```
wc-ui-developer:
  - slug:               <slug>
  - viewport:           <WxH>
  - pixel diff before:  <N>%
  - pixel diff after:   <N>%
  - patches applied:    <short bullets>
  - framework gaps:     <none | list>
  - report:             .audit/wc-ui/<slug>/report.md
```
